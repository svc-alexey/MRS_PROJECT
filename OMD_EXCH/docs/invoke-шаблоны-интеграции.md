# Шаблоны интеграции и endpoint `/invoke`

Документация по функциональности **единого HTTP-метода вызова логики интеграции** по описанию из справочника `ОДМ_МетодыИнтеграции` и процедуре `ОДМ_ОбменДанными.ВыполнитьПоШаблону`.

## Назначение

Вместо добавления отдельных URL под каждый сценарий внешняя система может вызывать один адрес **`POST …/invoke`**, передавая в теле JSON:

- какой сценарий выполнить (**код шаблона** в справочнике);
- произвольные данные (сырой JSON: даты, коды справочников, массивы строк табличной части и т.д.).

Обработка этих данных (маппинг в реквизиты документа, поиск ссылок, заполнение обязательных полей) задаётся в конкретной информационной базе в элементе справочника или во внешнем коде, который вызывается из шаблона.

Расширение содержит **механизм**; **элементы шаблонов** создаются и заполняются в каждой базе отдельно.

## HTTP-сервис

| Компонент | Расположение |
|-----------|----------------|
| Описание сервиса | `HTTPServices/ОДМ_xchg` |
| Обработчик | функция `invokePOST` в модуле HTTP-сервиса |
| Шаблон URL | `/invoke`, метод **POST** |

Корневой URL публикации задаётся конфигурацией публикации (как и для остальных методов этого сервиса).

Поток обработки: десериализация тела запроса в JSON → формирование `ОписаниеЗадачи` с полем `Метод = "invoke"` → проверка **характеристик доступа** → вызов `ОДМ_ОбменДанными.ВыполнитьПоШаблону(ОписаниеЗадачи)` внутри общего обработчика входящих задач.

## Тело запроса (JSON)

Обязательное поле верхнего уровня:

| Поле | Тип | Описание |
|------|-----|----------|
| **`Шаблон`** | Строка | **Код** элемента справочника `ОДМ_МетодыИнтеграции`, по которому ищется активная запись. |

Остальные поля — произвольные: они попадают в структуру **`ОписаниеЗадачи.Параметры`** после десериализации и доступны коду шаблона (см. ниже).

Пример:

```json
{
  "Шаблон": "СОЗДАТЬ_ЗАКАЗ_ПОСТАВЩИКУ",
  "Дата": "2026-04-28",
  "КодПоставщика": "SUP001",
  "Строки": [
    {"Артикул": "ТОВ001", "Количество": 10}
  ]
}
```

## Справочник `ОДМ_МетодыИнтеграции`

### Реквизиты

| Реквизит | Назначение |
|----------|------------|
| **Код** | Идентификатор шаблона; совпадает со значением `"Шаблон"` в JSON запроса. |
| **Наименование** | Человекочитаемое имя. |
| **Активен** | Если Ложь — вызов отклоняется с ошибкой. |
| **ТипШаблона** | Строка: один из значений списка выбора на форме (`ЗаписьОбъектов`, `Запрос`, `Обработчик`, `Составной`). Определяет сценарий выполнения. |
| **ОписаниеМетода** | Текстовое описание для администратора (контракт, примеры параметров). |
| **Шаблон** | Содержимое зависит от типа (см. таблицу ниже). |

### Табличные части

- **ВходныеПараметры** — документирование ожидаемых полей JSON (`ИмяПараметра`, `Тип`, `Обязательный`, `Описание`). На выполнение не влияют, только для формы и регламентов.
- **ВыходныеПараметры** — то же для возвращаемых данных.

## Типы шаблонов и поле «Шаблон»

| ТипШаблона | Содержимое реквизита **Шаблон** | Поведение |
|------------|----------------------------------|-----------|
| **ЗаписьОбъектов** | Необязательный фрагмент BSL **предобработки** | Если текст не пустой — выполняется через `Выполнить(...)` для подготовки данных под обмен; затем вызывается `ЗаписатьОбъект(ОписаниеЗадачи)` (как для `/object`). |
| **Запрос** | Текст запроса на языке запросов 1С | В `ОписаниеЗадачи.Параметры` подставляется `ТекстЗапроса` из справочника; остальное тело запроса используется как параметры запроса обмена → `ВыполнитьЗапрос`. |
| **Обработчик** | Произвольный BSL | Выполняется через `Выполнить(...)`. Обычный сценарий: полный контроль — создание/запись объектов, запросы, формирование `ОписаниеЗадачи.Результат`. |
| **Составной** | Зарезервировано | Не реализовано; вызов завершается ошибкой. |

## Как из шаблона получить доступ к данным из JSON

После разбора тела запроса все корневые поля JSON доступны в **`ОписаниеЗадачи.Параметры`**.

Код реквизита **Шаблон** выполняется через встроенную функцию **`Выполнить`** в контексте процедуры **`ВыполнитьПоШаблону`**, поэтому в этом коде видны локальные переменные процедуры, в том числе **`ОписаниеЗадачи`**.

Типичный приём в коде типа **Обработчик** или в предобработке **ЗаписьОбъектов**:

```bsl
Парам = ОписаниеЗадачи.Параметры;
// Парам.КодПоставщика, Парам.Строки, ...
```

Итог операции для ответа вызывающей стороне задаётся через **`ОписаниеЗадачи.Результат`** (и стандартную обработку задачи обмена), как для других методов HTTP-сервиса.

## Доступ и безопасность

- Перед выполнением вызывается **`ХарактеристикиДоступа`**: для клиента должно быть разрешение на сервис, соответствующий методу **`invoke`** (аналогично `query`, `objectPOST` и т.д.) — настройка в справочнике баз данных интеграции (`ОДМ_БазыДанных`, табличная часть разрешений).
- Типы **Обработчик** и предобработка **ЗаписьОбъектов** используют **`Выполнить`**: это эквивалент выполнения произвольного кода на сервере. Шаблоны должны создаваться только доверенными администраторами; внешним системам выдавать доступ к `/invoke` следует осознанно.

## Связанные объекты кода

| Объект | Назначение |
|--------|------------|
| `ОДМ_ОбменДанными.ВыполнитьПоШаблону` | Поиск элемента по коду, проверка активности, ветвление по `ТипШаблона`. |
| `ОДМ_ОбменДанными.ЗаписатьОбъект` | Запись объекта обмена из параметров (после опциональной предобработки). |
| `ОДМ_ОбменДанными.ВыполнитьЗапрос` | Выполнение запроса с параметрами из `ОписаниеЗадачи`. |
| HTTP-сервис `ОДМ_xchg`, функция `invokePOST` | Точка входа `/invoke`. |
